Build Report: 48_hopper_warp_specialized_gemm on NVIDIA Thor (SM 11.0)

This report documents the exact steps taken to build and run the CUTLASS example 48_hopper_warp_specialized_gemm on an NVIDIA Thor GPU (SM 11.0) using CUDA 13.1 and CMake 3.28. The example was originally written for Hopper (SM 90) and required source-level adaptations and a custom build command to run on Thor.

Why the Example Cannot Be Built Naively

Three independent problems prevent a straightforward build-and-run on SM 11.0:

#ProblemRoot cause
1 CMake 3.28 generates compute_1100 (four digits) when CUTLASS_NVCC_ARCHS=1100 is passed, which NVCC 13.1 rejects. CMake 3.28 does not know the SM 11.0 naming convention; it zero-pads the arch number.
2 The example uses SM 90 GMMA (wgmma.mma_async) instructions, which are not available on SM 11.0 (Thor uses TCGEN05 / UMMA instead). SM 90 and SM 11.0 have incompatible MMA instruction sets; SM 11.0 is not a superset of SM 90.
3 SM 11.0 cannot run SM 90a or SM 100a cubins, and PTX JIT from those targets fails with "unsupported toolchain" or "no kernel image available". The CUDA driver on this system cannot JIT-compile PTX generated by NVCC 13.1 for earlier architectures onto SM 11.0.
Solution summary: bypass CMake's arch handling entirely, adapt the source to use the SM 100 (Blackwell) CUTLASS kernel path (which SM 11.0 supports natively via TCGEN05), and compile directly with nvcc targeting sm_110a.

Step-by-Step Build Instructions

1 Verify the environment

Confirm the GPU architecture and CUDA version before proceeding.

nvidia-smi --query-gpu=name,compute_cap --format=csv,noheader
# Expected: NVIDIA Thor, 11.0

nvcc --version
# Expected: release 13.1 or newer

2 Create the build directory

The CMake configure step is still useful for generating the full project structure and resolving include paths, even though we will not use its generated build rules for this specific example. 

mkdir -p build
cd build

cmake ../cutlass \
  -DCMAKE_BUILD_TYPE=Release \
  -DCUTLASS_NVCC_ARCHS="90a" \
  -DCUTLASS_ENABLE_EXAMPLES=ON \
  -DCUTLASS_ENABLE_TESTS=OFF \
  -DCUTLASS_ENABLE_TOOLS=OFF
Note: CMake will emit errors about empty CUDA_ARCHITECTURES for some examples (e.g. 113_hopper_gemm_activation_fusion) because CMake 3.28 does not recognise 90a as a valid CUDA arch. These errors do not affect the example we are building — the configure step still writes the necessary CMakeCache.txt and include-path information.

3 Patch the source file for SM 11.0

The original example targets SM 90 (Hopper). Seven changes are needed to redirect it to the SM 100 (Blackwell) CUTLASS kernel path, which SM 11.0 supports natively. The original file is preserved as 48_hopper_warp_specialized_gemm.cu.bak. 

# Back up the original
cp cutlass/examples/48_hopper_warp_specialized_gemm/48_hopper_warp_specialized_gemm.cu \
   cutlass/examples/48_hopper_warp_specialized_gemm/48_hopper_warp_specialized_gemm.cu.bak

Why these changes?
  • ArchTag / guard macros: Sm100 selects the Blackwell (TCGEN05 / UMMA) kernel path in CUTLASS's CollectiveBuilder, which SM 11.0 supports natively. There is no Sm110 arch tag in this CUTLASS version.
  • TileShape / ClusterShape: SM 100 TCGEN05 MMA operates on a 256×128×64 tile across 2 SMs; the SM 90 128×128×32 tile is specific to GMMA and is invalid here.
  • 4-D ProblemShape: GemmUniversal with an SM 100 mainloop requires an explicit batch dimension.
  • kernel_hw_info / raster_order: the SM 100 scheduler does not expose raster_order in its Arguments struct, and kernel_hw_info is not needed for the SM 100 adapter.
  • Runtime check: accept any GPU with major ≥ 10 (SM 100, SM 110, …).

4 Compile directly with nvcc

CMake 3.28 cannot generate correct --generate-code flags for SM 11.0 (it produces the invalid compute_1100). We therefore invoke nvcc directly, targeting sm_110a — the "all-features" variant of SM 11.0 that enables __CUDA_ARCH_FEAT_SM110_ALL and thus activates TCGEN05 and TMA inside CUTLASS's device-side guards. 

# Set paths
CUTLASS_DIR=/home/leimyu/01_research/cutlass_dsmem/cutlass
EXAMPLE_DIR=$CUTLASS_DIR/examples/48_hopper_warp_specialized_gemm
OUT_DIR=/home/leimyu/01_research/cutlass_dsmem/build/examples/48_hopper_warp_specialized_gemm
mkdir -p $OUT_DIR

# Compile
nvcc \
  -std=c++17 \
  -O3 \
  --generate-code=arch=compute_110a,code=[sm_110a,compute_110a] \
  --expt-relaxed-constexpr \
  -lineinfo \
  -ftemplate-backtrace-limit=0 \
  -DCUTLASS_ENABLE_TENSOR_CORE_MMA=1 \
  -DCUTLASS_ENABLE_GDC_FOR_SM100=1 \
  -I$CUTLASS_DIR/include \
  -I$CUTLASS_DIR/tools/util/include \
  -I$CUTLASS_DIR/examples/common \
  -I$CUTLASS_DIR/examples/48_hopper_warp_specialized_gemm \
  $EXAMPLE_DIR/48_hopper_warp_specialized_gemm.cu \
  -o $OUT_DIR/48_hopper_warp_specialized_gemm \
  -lcuda

Key flags explained:

FlagPurpose
--generate-code=arch=compute_110a,code=[sm_110a,compute_110a] Compile a native SM 11.0a cubin and embed PTX for forward compatibility. The 110a variant sets __CUDA_ARCH_FEAT_SM110_ALL, which activates TCGEN05 MMA and TMA device-side guards inside CUTLASS headers.
--expt-relaxed-constexpr Required by CUTLASS's heavy use of constexpr in device code.
-DCUTLASS_ENABLE_TENSOR_CORE_MMA=1 Enables tensor-core MMA paths in CUTLASS (required for SM 100/110 UMMA).
-DCUTLASS_ENABLE_GDC_FOR_SM100=1 Enables Grid Dependency Control (GDC / programmatic dependent launches) used by the SM 100 persistent tile scheduler.
-I$CUTLASS_DIR/examples/common Provides helper.h (GPU timer, CUDA/CUTLASS error-check macros).
-lcuda Links the CUDA driver library (needed for TMA descriptor APIs).
Why not sm_100a? SM 100a cubins cannot run on SM 11.0 — the CUDA driver returns "no kernel image is available for execution on the device". SM 11.0 (Thor) and SM 10.0 (Blackwell) are distinct architectures; SM 11.0 is not a superset of SM 10.0 at the binary level. Compiling natively for sm_110a is the only correct approach.

5 Run the example

# Default problem size (5120 × 4096 × 4096)
$OUT_DIR/48_hopper_warp_specialized_gemm

# Custom size with rasterization and swizzle options
$OUT_DIR/48_hopper_warp_specialized_gemm \
  --m=2048 --n=2048 --k=2048 \
  --rasterization=N --swizzle=2

Architecture Compatibility Notes

Target arch Runs on SM 11.0? Reason
sm_90 / compute_90 ✗ PTX JIT fails NVCC 13.1 PTX is rejected by the driver as "unsupported toolchain"
sm_90a / compute_90a ✗ No kernel image SM 90a cubin is not binary-compatible with SM 11.0
sm_100 / compute_100 ✗ PTX JIT fails Same toolchain mismatch as SM 90
sm_100a / compute_100a ✗ No kernel image SM 100a cubin is not binary-compatible with SM 11.0
sm_110 / compute_110 ✓ Runs, but CUTLASS guards inactive Native SM 11.0 cubin, but __CUDA_ARCH_FEAT_SM110_ALL is not set so TCGEN05 / TMA device-side guards in CUTLASS headers are disabled, causing assertion failures inside the SM 100 kernel.
sm_110a / compute_110a ✓ Works correctly Native SM 11.0a cubin; sets __CUDA_ARCH_FEAT_SM110_ALL, enabling all TCGEN05 and TMA guards. This is the correct target.
