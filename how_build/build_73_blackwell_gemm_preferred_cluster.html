Build Report: 73_blackwell_gemm_preferred_cluster on NVIDIA Thor (SM 11.0)

This report documents building and running cutlass/examples/73_blackwell_gemm_preferred_cluster/blackwell_gemm_preferred_cluster.cu on an NVIDIA Thor GPU (SM 11.0) using CUDA 13.1, placed in the shared build_sm110/ tree alongside examples 48 and 70.

This example demonstrates Preferred Dynamic Clusters introduced in CUTLASS 3.8 for SM 100+: the ability to specify a preferred cluster shape and a fallback cluster shape at runtime, so the scheduler can use the larger cluster where SM occupancy allows and fall back to the smaller one for the remainder.

What This Example Does

The kernel uses a dynamic cluster shape (ClusterShape_MNK = Shape<int,int,_1>) so that the M and N cluster dimensions are set at runtime rather than baked into the template. Two shapes are passed via arguments.hw_info:

  • Preferred cluster — the larger shape the scheduler tries first (e.g. 4×4×1).
  • Fallback cluster — the smaller shape used for leftover SMs (e.g. 2×1×1).

The kernel schedule is explicitly set to KernelTmaWarpSpecialized2SmSm100 / TmaWarpSpecialized2Sm (2-SM cooperative MMA) because KernelScheduleAuto would conservatively select 1-SM MMA when it sees a dynamic cluster shape.

Cluster shape constraints enforced by can_implement():
  • Preferred cluster depth (Z) must equal fallback cluster depth.
  • Fallback shape must evenly divide preferred shape.
  • Because the kernel uses 2-SM MMA, the cluster M dimension must always be a multiple of 2 — both preferred and fallback.

Why It Cannot Be Run Naively on SM 11.0

#ProblemRoot cause
1 Runtime check props.major != 10 || props.minor != 0 rejects SM 11.0. Guard written for Blackwell SM 100 only; Thor has major=11.
2 CMake 3.28 cannot generate correct arch flags for SM 11.0. Same zero-padding issue as examples 48 and 70; bypassed by invoking nvcc directly.
The CUDA version check (< 12.8) and all kernel template types (Sm100, Shape<int,int,int,int>, dynamic cluster shape, KernelTmaWarpSpecialized2SmSm100) are already correct for SM 11.0 — no template changes are needed.

CUTLASS_DIR=/home/leimyu/01_research/cutlass_dsmem/cutlass
BUILD_DIR=/home/leimyu/01_research/cutlass_dsmem/build_sm110

nvcc \
  -std=c++17 -O3 \
  --generate-code=arch=compute_110a,code=[sm_110a,compute_110a] \

  -lineinfo \
  -ftemplate-backtrace-limit=0 \
  -DCUTLASS_ENABLE_TENSOR_CORE_MMA=1 \
  -DCUTLASS_ENABLE_GDC_FOR_SM100=1 \
  -I$CUTLASS_DIR/include \
  -I$CUTLASS_DIR/tools/util/include \
  -I$CUTLASS_DIR/examples/common \
  -I$CUTLASS_DIR/examples/73_blackwell_gemm_preferred_cluster \
  $CUTLASS_DIR/examples/73_blackwell_gemm_preferred_cluster/blackwell_gemm_preferred_cluster.cu \
  -o $BUILD_DIR/examples/73_blackwell_gemm_preferred_cluster/73_blackwell_gemm_preferred_cluster \
  -lcuda

3 Run

BIN=$BUILD_DIR/examples/73_blackwell_gemm_preferred_cluster/73_blackwell_gemm_preferred_cluster

# Default: preferred=4x4, fallback=2x1
$BIN

# Larger problem, preferred=4x2, fallback=2x1, swizzle=2
$BIN --m=8192 --n=8192 --k=4096 \
  --preferred_cluster_m=4 --preferred_cluster_n=2 \
  --fallback_cluster_m=2  --fallback_cluster_n=1 \
  --swizzle=2

# Smaller preferred cluster: preferred=2x2, fallback=2x1
$BIN --m=4096 --n=4096 --k=4096 \
  --preferred_cluster_m=2 --preferred_cluster_n=2 \
  --fallback_cluster_m=2  --fallback_cluster_n=1
